App Server: Client/Server Convergence
on the Web

By John Hoagland, Components Online (draft)

The App Server market is a result of an unprecedented convergence of client/server techologies caused by the web. For the related practice of offering desktop and other applications as services via the web as well as the evolution of this market, see Web Server--Key Technology as New Kind of App Server.

Server Software Being Transformed. Traditional enterprise server software is rapidly migrating to the web. While the database itself is typically left on its own "tier," the server software that adds every kind of functionality is moving to the web server as this new kind of "middle tier" application server.

This is not vaporware, but is being driven by market forces partly to aid high-end web sites in handling immense volumes of client contacts via the browser.

Literally billions of dollars are being transacted over the web in e-commerce ranging from Amazon books to Cisco routers. The high dollar amounts are business to business transactions where supplier and customer set up customized ordering web pages tailored to the relationship. Databases automatically handle this commerce on the back end. Once in the database of course, further "massaging" of the data is endless.

"Application server" is the term of choice for providing the web server software infrastructure for web authoring/ programming on the "middle tier" that the web server represents. See Application Servers: Behind the Mystery, by Nelson King, Intelligent Enterprise February 16, 1999.

Sometimes the web server itself is called middleware, as the middle tier in "n-tier" architectures, but "middleware" is more specifically reserved for the "plumbing" or software infrastructure used on the web server. Middleware thus encompasses both infrastructure associated with OSes, in this case the NOS, such as security services.

In addition, specific middleware services such transaction processing are so universal that these can be thought of as "middleware," although generally this functionality is catalogued as a kind of server application. See Ecommerce Transactions for a detailed view of transaction processing in the middle tier.

Of course, the universal availability of quick and easy web authoring drives this process on the client side. Specialized "programmers" are not needed in most cases.

Server Side "Programming."

On the server side, the relative obscurity of little known programs like CGI and Perl, which carried the load of forms processing in the early stages of web development, helped to hide the key importance of web server software.

Now the shift to the server side is the inevitable effort to remedy the primitive state of web server software in facilitating and supporting these transactions in particular and web applications in general. The dream of Java applets being downloaded on the client side to do the heavy lifting has never really materialized. Instead Java as the language has gained new life on the server side, but is subject to immense competition, especially from componentized systems of its own and otherwise as we shall see.

The success of a different ASP (Active Server Pages instead of Application Service Provider) acronym has led to a Java version in JSP (Java Server Pages).  ASP and JSP allow web page programming to "call" functionality on the server so programs can be executed on the server, but returned as garden variety HTML web pages.

XML ([X]etensible Markup Language), a variation of HTML that works with HTTP, sticks DB fields through sets of documents to create a kind of data exchange format between HTML files. Now SOAP (Simple Object Access Protocol), a means of calling objects (components) or programs via XML (and HTTP), is proposed to give web pages greatly expanded programability on both client and server.

These technologies increase the integration of the web tier with traditional web server software, such as Online Analytical Processing (OLAP), that includes everything from Enterprise Resource Planning (ERP) to data mining and smart agents.

Web Integration: What it Means

However, the emergence of the web server as the key technology is a result of deeper trends. The web is forcing an integration far more profound than appears on the surface.

The mass market created by the web itself has changed the economics of the web server. Just as the $100,000 pilot training technology may end up in the mass produced joystick, the web server mass market encourages the investment in basic server technologies such as transaction processing to pay off in the mass quantities sold.

Today, every small or home business can rent a site on a web server. This mass market creates a greater potential revenue stream than even the biggest corporation standing alone. The fear of these enterprise giants is the displacement by the more energetic of small companies in the 20-50 employee range where NT Server is targeted. Such a company with a web business plan coupled with a IPO can be a fearsome competitor in a world where Yahoo is capitalized to a greater extent than major oil companies.

What appears to be happening is an integration of the application itself. In terms of server software, the result is a kind of "super-app" such as MTS, where traditional, very expensive, server applications are being built into the plumbing.

This is starting to happen in important sectors of the economy where a service extensible model like COM allows ISPs to create the web infrastructure for these sectors. See Hot Trend: Interoperable component suites dominate manufacturing production software, Start Magazine, March 1, 1999 and Microsoft plans architecture for manufacturing companies, by Bob Trott
InfoWorld Electric, February 20, 1999.

These companies reside higher in the food chain than the truly interchangeable software parts, or components, implementing the infrastructure code in particular instances. This code can be expected to be a commodity. Copyright protection is not expected to extend to the interoperable interfaces of this infrastructure.

Prices can be expected to drop dramatically, while at the same time, vast new markets open up. The player who becomes established in providing the application server infrastructure leading category product may become the equivalent on the server to the Lotus spreadsheet client application product.

The industrial age equivalent might be a whole technology/product category, such as the radio or the automobile. Considering that it is estimated that 60% of the GNP is related to office-while collar tasks (against 5% re entertainment vis a vis the TimeWarner/AOL merger), then the implications of a successful product in the application server arena is enormous.

Successful products will be the infrastructure or combination of components that make up a product. Now called ASP for Application Server Providers, a likely morph will occur into Application Server Products, even though the conventional wisdom is that everything will flow to the server side.

Why? The same IT bottleneck developing around the very provision of these services or "servers."

The component, or interchangeable software part, creates the technological imperative to build these systems, just as the interchangeable hardware part has been at the center of our industrialized economy. A similar software industrialization is both a result of the power provided by componentized software and a cause of its success.

In this way, the application server provides the componentized environment of application services necessary to build the web server infrastructure for, and out of, components, the basic building block of this new technological age.
