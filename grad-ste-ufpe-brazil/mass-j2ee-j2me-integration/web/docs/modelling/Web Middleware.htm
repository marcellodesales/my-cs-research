Web Middleware Explained

By John Hoagland, Components Online (draft)

Web middleware is basically client/server middleware. However, the web infrastructure greatly simplifies its use.

We will look at the basic technologies integrating the internet and the computer. Then we will look at web systems of increasing complexity from the Hypertext Era of browsers connected to web servers via HTTP, to the Interactive Era of  forms querying databases via CGI, to the Distributed Object Era of components communicating with app servers via ORBs.

The Client. The "client" is a term for the ordinary desktop computer used when the desktop is referenced in terms of its role in a network. Basically the client receives a service from another computer we call a server.

The Server. Similarly, a server is a computer that supplies a service to the client over a network.

Middleware. Middleware has been called the "/" or "plumbing" in client/server. It is the set of technologies that support the interaction providing a "service" to the client from the server over a network.

NOS, or Network Operating System. The NOS is an operating system that has built in network capabilities. These days there is no clear distinction between computers that are clients or servers. If there is a network link and service software on the computer, contemporary computers have the power to function as servers. Many of the early intranets were run on workstations. Generally however, desktops like Windows 9x, NT Workstation, Mac, OS2 and Linux function at the lower end of the scale.

Unix, NT (now Windows 2000) Server, Netware, Linux, and mainframe MVS dominate the high and middle end of the NOS market.

TCP/IP, the Internet Protocol. TCP/IP means Transmission Control Protocol/Internet Protocol. Developed by the Defense Department's Advanced Research Projects Agency (ARPA) for robust transmissions even if part of the network goes down in war, the core Internet Protocol (IP) features transmission by packets and a true internetworking system.

IP packets have header and data parts analogous to the instruction and data parts of the computer's machine code train, except that the IP packet is destined instead for a network of computer routers and bridges, rather than the computer's internal CPU's "network" of registers.

The transmission of packets are ideally suited to computers. Large data packets are fragmented and reassembled, a methodology ideally suited to the computer which parses in nanoseconds.

The basic difference between a phone/fax circuit, which is held open, and the same message that is run over the internet in packets is that the internetworking technology is 10,000 times shorter and cheaper. In other words, the computer operating in nanoseconds can send umpteem messages while a caller is saying, "um."

The Internet Protocol is designed as a true internetworking layer--hence the name internet. IP, by a header tag, runs on a wide variety of underlying physical networks, called the Network Interface Layer, including LANs (Local Area Networks) such as Ethernet and Token Ring, or WAN (Wide Area Networks), such as X.25, Frame Relay or the upcoming ATM.

TCP (along with the rare UDP), called the Transport Layer, ensures host-to-host reliability and provides flow control management.

On top of TCP/IP, is the Application Layer for running applications such as FTP, SMTP, POP, HTTP, etc. Each application distinguishes itself by its TCP port number. We look at HTTP in greater detail here.

HyperText Era. The initial web era involved the browser simply calling web pages or files written in HTML down from a web server using the http protocol. This era is also characterized by messaging oriented middleware (MOM), supporting the soon-to-be ubiquitous email.

 TCP/IP Frequently Asked Questions by Mike Oliver.

The Browser. If any one item is associated with the internet, it is the web browser. A GUI (Graphical User Interface) based program, the browser basically retrieve files written in HTML (HyperText Markup Language) over the internet, using the HTTP protocol.

The web browser, called a thin client, because the browser is in essence a 3270 terminal from the mainframe days.

The thin client is juxtaposed with the fat client, which refers to the current desktop computer with its rich functionality.

Web Servers. In the hypertext context, web servers are often called hosts for their function of hosting and serving up web pages. But web servers also link forms on the client side to databases on the server side.

In this interactive context, web servers function as another form of middleware--as the middle tier in three-tier, or n-tier, systems.

HTTP, the Hypertext Protocol. Built on top of the TCP/IP protocol as one of its applications, HTTP is a kind of RPC (Remote Procedure Call).

What distinguishes this protocol is the use of hyperlinks to pull files (web pages), by simply clicking on a hypertext link, representing a universal addressing system, called a URL (Uniform Resource Locator) from remote hosts on web servers.

A simple page markup language, HTML (HyperText Markup Language), enables the universal creation of web page, by HTML editors or web authoring programs that are as easy to use as word processors.In addition, the graphical images may be easily inserted in web pages.

 "The Navigator is more like a 3270 terminal than anything else, with the big difference being that you don't just use that terminal to access one mainframe." Marc Andreessen, VP Netscape, The Essential Client/ Server Survival Guide, Second Edition, p. 488.
MOM (Message Oriented Middleware). MOM, the basis of email, is the other fundamental way to communicate over a wire. While RPC requires a ongoing or synchronous link, MOM can get by on an intermittent link or asynchronous link. MOM stores things in "lines" or queues. MOM- based systems scale much more effectively.

MOM is the basis of email and groupware, a somewhat nebulous collection of collaborative related functionality, including conferencing, personal information managers (PIMs), document management and workflow. The two most important groupware products are IBM/Lotus's Notes and Microsoft's Exchange.

Interactive Era. The interactive era allows the user to make entries in forms embedded in web pages. The response is information generated from databases linked to the web server on the back end, utilizing methodogies such CGI and ASP.

Forms. Forms have a rich history as the user interface for complex interactions with databases. Underlying the form elements can be a intensive programming regime. The GUI (Graphical User Interface), along with a myriad of dialogue boxes, can be throught of as a "forms" method for an app to interact with the underlying OS. New Web forms are coming, Daily Geek News, UGeek, July 10, 2000.
CGI. Originally, web interactive forms used CGI (Common Gateway Interface) along with scripting languages such as JavaScript (the standard on the client side). Now the industry is moving rapidly toward a distributed component context. The Common Gateway Interface, NCSA HTTPd
ASP (Active Server Pages). ASP, as well as JSP (Java Server Pages) and the open source PHP, strips web page "code" and executes it on the server, before returning it as an ordinary web page that can run on any browser.

The significance of ASP, JSP and PHP is that they potentially expand markets greatly by allowing web page authors to "pierce" the middle tier "veil." This leads the way for ordinary users to participate in middle tier functionality with upcoming desktop apps.

For ASP+, see .NET Platform as Component Infrastructure.

ASP should not be confused with the evolving use of this acronym for Application Service Providers.

 Evolution of a Super Hero, by Tobias Ratschiller, Web Techniques, April 2000.
The Database. The third, or back end, tier, in terms of three tier infrastructure, is the database. Web servers easily incorporate existing server-to-server technologies to link to databases, such as the universal SQL (Structured Query Language) and ODBC (Open DataBase Connectivity) as well as the newer JDBC (Java DataBase Connectivity) and Microsoft's ADO (Active Data Objects).

Databases are involved with two major technologies, transaction and analytical processing. In terms of the internet, these are called OnLine Transaction Processing (OLTP) and OnLine Analytical Processing (OLAP). Transaction processing is operational use of databases which we will discuss under App Servers.

OLAP is separate "decision support" processing that appears in many contexts, often appearing as data warehouses, but is migrating rapidly to the web.

Data warehouses are collections of now "historical" information from operational databases and a variety of others sources. These were coupled with a variety of analytical tools.

Datamarts were specialized data collections related to specific business areas.

Insight

Erik Tompsen, in Through the Looking Glass, suggests that these same DB technologies underlie internet "portals," the web version of the corporate data warehouse, with some of the same unrealized promises.

Data mining and personal information agents were frequently added to the innovation mix, coupled with data visualization aids to aid the user in understanding the results.

The organizations that succeeded in the information age built their business intelligence strategies on top of a fully integrated data warehousing foundation. One of the major benefits of data warehousing that was driving significant ROI was data mining. Many enterprises were utilizing data mining to reveal trends, explain known outcomes, predict future outcomes and identify factors that could lead to a desired outcome. These companies gained the ability to make better business decisions, increase revenue and improve competitive advantage. Data mining provided insights that were not readily apparent in large volumes of data by utilizing intelligent agents, many of which were developed in the mid-eighties as a result of artificial intelligence (AI) discoveries. Savvy corporations used data mining to develop marketing strategies, target mailings, adjust inventories, minimize risks and eliminate wasteful spending based on an analysis of their data. As data visualization software evolved, it made the results of data mining more understandable to the end users, eliminating the need for a statistician. See A 10 Year Journey.

Shared by all is a forced march to the web. But the web itself is undergoing intense change.

Distributed Object Era. On top of the technological change spawned by the web, the era of distributed objects or components is just beginning. While components themselves have emerged as a mature technology, and often the only economic solution, the use of components on the web has depended on overcoming the "dumb terminal" era of the web, via XML and SOAP.
Components. As we elucidated in What are Components?, the binary standard of components mean that they are designed to interop across networks. In fact, the true encapsulation of interface-based components is especially mandated by distributed systems like the web.

With their interfaces fixed, components become the basis of interchangeable software parts, with reuseable functionality as commodities plugged in to their app server infrastructure.

Components have been variously defined as user-oriented and client-oriented software.

While components may be found on all tiers, it is the client side which is in the greatest need of interoperability of the web client and the underlying desktop functionality where the most basic component forms, compound documents, can be integrated via the web container to be capable of being distributed. See Software Machines: Integrated Browser.

The achievement of economies of scale and user-based plug-in programming are the next stages of a new software- and communciations- based industrial age. Presently, we are like the pre-industrial people who had to depend on custom-based, craft workers, i.e. programmers, for products in mass quantities and functionality.

App Servers. The application server is a generic category for server software that provide a middle-tier service infrastructure for componentized systems, especially that related to ecommerce. See App Server: Client/Server Convergence on the Web.

For instance, transaction processing (TP) monitors support component-based solutions by providing services such as ACID properties. See Ecommerce Transactions for a detailed view of transaction processing in the middle tier.

Components, due to their specialized functionality, fit the bill because they can function as stateless object with the data sent by MOM means. Consequently, the software system scales effectively. See Economic Transactions: TP Layer.

But the convergence in this field really represents a rethinking of whole systems in much the way the original systems analysis began to streamline processes in the nascent computer age by simply looking at all a company's forms and seeing how they interrelated.

There is a lot of confusing nomeclature here as the generic words server and application have different connotations. However, due to the web convergence, no one is certain how definitions sort out.

Important to some commentators is the means of delivery, i.e. a service, such as provided by an Application Service Provider (ASP), rather than a product, such as an Application Server Product, often called App Servers. The latter are the original enterprise server software vendors who focused on the analysis of data.

While OLTP provides the basic processing of online transactions, OLAP (OnLine Analytical Processing) supplies the analysis of what these transactions mean for a company based on the historic data.

Kinds of analysis include ERP (Enterprise Resource Planning), CRM (Customer Relations Management) and SCM (Supply Chain Management).

OLAP, strictly speaking, involves multidimension tables and cubes, a "narrow" kind of database middleware for fast, efficient processing of data, often stored in the data warehouse, an overall conceptual scheme of integrating data, as well as the more functionally delimited data mart.

But OLAP can be broadly describe all of the non-operational data processing done for analytical purposes. Other processing tools include data mining and smart agents as well as many other product areas that serviced the enterprise.

The process of turning data into knowledge and knowledge into action for business gain became known as business intelligence. It was an end-user activity that was facilitated by various analytical and collaborative tools and applications as well as a data warehousing infrastructure. It encompassed all types of data including hierarchical, relational, text, spatial, audio and video. Knowledge management included the extraction, documentation and cataloging of unwritten business rules and contextual information based on experiences that individuals "stored" in their minds. Business intelligence focused on how organizations accessed and used data to achieve organizational objectives. Business intelligence tools included ad hoc query and reporting tools, managed query environments, online analytical processing (OLAP), data mining and data visualization. Access to information, regardless of location, was a must in the data-driven business world. Data replication made it possible for remote locations to access up-to-date information at any time. As business intelligence continued to evolve, the integration of query, reporting and OLAP into a single product became critical for the end user. The complete business intelligence suite started to become a reality. The introduction of commercially available natural language query products that converted the English language into SQL queries offered the promise of eliminating the need for end users to understand the syntax of SQL.  See A 10 Year Journey.

Customer support, called Customer Relations Management (CRM) became a popular focus. From the corporate side, data visualization tools have been supplemented by user-based data reporting tools.

From the consumer side, a data warehouse may become quite intricate when put online as a "portal" to service the customer, such as online stock trading site that includes numerous analytical tools.

Yet widespread dissatisfaction is expressed on the part of the customer or user, reminescent of the same problems with centralized EDP that preceded the PC revolution. Only the choice from a wide assortment of programs at many levels targeted to the end user will succeed.

And even that may not suffice.

Coming from the desktop arena are the software forces of the mass market from the PC industry. This part of the industry is keenly attuned to the end user, having engineered the applications that made the PC revolution possible that defeated the earlier version of a centralized server-side approach.

XML. The web page increasingly functions as a universal container. In addition, XML (Extensible Markup Language) provides a universal data formatting standard embedded in web pages for structuring pages as data elements within the context of the W3C's DOM (Document Object Model).

XML is regarded as a furturistic ASCII, a way of using tags to abstract the key string elements of text across heterogeneous computers. A XML parser classifies the text's character string into a info or dataset of elements and attribures by a type system or schema.

So the structure itself is a machine readable database where the "metadata" tagged fields are the document "type-definded" elements and the content description are their values. XML can be thought of as functioning like a file tree structure, except that the structure goes to individual web pages as objects or components.

"For example, to get stock portfolio data from a Web-based database and deliver it securely to Excel for analysis, the database server returns a Recordset to an IIS ASP page. The ASP converts the Recordset to an XML stream and sends it over the wire. Excel VBA code retrieves the ASP page content from the Web page and a function converts the XML stream back into a Recordset to be consumed by Excel [note the Recordset is being subsumed by the .NET Dataset]." The Next Contender: Com+ .

There are all sorts of XML "description" languages or schemas for "database" markup languages, such as Web Services Description Language (WSDL).

For the internet, XML supercedes EDI (Electronic Data Integration) because XML is the universal data format sanctioned by the W3C.

As an "extensible" model, XML is a standard basis for setting data format standards. However, the universal nature of the web page as a component-based container gives XML its own strength as a universal data formatting markup language.

SOAP. One kind of content proposed to be embedded in the XML data structure is method calls and event messages for objects or components, such as the W3C's SOAP (Simple Object Access Protocol).

SOAP is a minimal distributed component infrastructure that supercedes DCOM and CORBA in the HTTP arena. See A Young Person's Guide to The Simple Object Access Protocol: SOAP Increases Interoperability Across Platforms and Languages.

SOAP is a "protocol that defines how to package a method request as an XML string that can be delivered over HTTP. SOAP is rapidly becoming the lingua franca for eCommerce." See SOAP. An overview of the Simple Object Access Protocol.

Basically, SOAP breaks down complex data types for network transmission and reconstitutes them on the other end of the wire. See SOAP for Platform-Neutral interoperability.

The combination of XML and SOAP means that the client computer would, once again, become much more than a terminal as the great functionality of client applications could be integrated with the web's data sources:

". . . it's sort of a dumb terminal world the way it's being done today. Imagine the next generation, which is a programmatic way to use the data that's sent to you. Certainly XML is a key building block for doing that, but that's insufficient. You need rich schemas and you need the ability to have a programming model based on those schemas. Imagine you had an auction site and a banking site, and they each exposed the schemas necessary so that when the page comes back to the client, you could go and process the schemas so you could make programmatic decisions easily on the client. When an auction item hits a certain price, you could transfer money from your bank account over into the auction automatically." Microsoft's Jim Allchin interviewed by InfoWorld.

The Platform or ORB (Object Request Broker). Platforms or ORBs provide the basic "plumbing" specs for the client/server componentized infrastructure for components by providing the standard or model by which systems are developed. Each ORB is a model that specifies the protocols or ways to "package" interactions and system services such as security and load balancing provided the underlying components.

The competing ORBs are CORBA (Common Object Request Broker Architecture), Microsoft's COM (Component Object Model) and .NET and Sun's J2EE, especially Enterprise JavaBeans. Generally, the term ORB has been replaced by the generic "platform."
