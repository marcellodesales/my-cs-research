Indexed Sequential Access Method
Avl and Red-Black trees guarantee good performance when the data is small enough to reside in memory all at once.
For larger dictionaries or files, that must reside on the disk we get improved performance using search tree trees of higher degree (more than 2 children)
First we will look at indeed sequential access method (ISAM).
Provides good sequential and random access
Disk is divided up into blacks
A block is the smallest unit that can be written to
Typically it is 1 track long and can be input or output with a single seek and latency delay.
The dictionary elements are packed into blocks in ascending order and the blocks are used in an order that minimizes the delay in going from one block to another.

More ISAM
For sequential access the blocks are input in order and the elements in each block are retrieved in ascending order.
If the block contains m elements, the number of disk accesses per element retrieved is 1/m.
To support random access a an index is maintained
This index contains the largest key in each block to search
To search for the element with key k, the index is for the single block that contains k
The entire block is retrieved and searched internally for the desired element
This can be extended to files that are larger than 1 disk
Now elements are assigned to disks and then to blocks
Each disk contains an index with the largest key in the block
First a disk index is searched to find the correct disk and …

m-way Search Trees
Definition. An m-way search tree may or may not be empty.
Each node has up to m children and between 1 and m-1 elements
Every node with p elements has exactly p+1 children
Consider any node with p elements. Let k1, …, kp be the keys of these elements.  The elements are ordered so that k1<k2<…<kp. Let c0,c1, …, cp be the p+1 children of the node. The elements in the subtree with root cp have keys smaller than k1, while those in the subtree with root cp  have keys larger than kp, and those in the subtree with root ci have keys larger than ki but smaller than ki+1, 1 < i < p.

Height of an m-Way Tree
 An m-way tree of height h has
at least h elements
At most mh-1 nodes
Hence if the tree has n nodes, its height ranges from n to logm(n+1)
A 200 way search tree of height 5 for example can hold 21*1010 –1 elements but might hold as few as 5.
Or a 200 way tree with 32*1010-1 elements has a height between 5 and 32*1010-1.
To get better performance we must have the tree more balanced.

B-Trees of Order 2 and 3
In a B-tree of order 2, no internal nodes has more than 2 children. Since an internal node must have at least 2 children, all internal nodes of a B-tree of order 2 have exactly 2 children.
The number of nodes in order 2 B-tree is 2h-1
A B-tree of order 3 has either 2 or 3 children.
It is also known as a 2-3 tree.
Height of a B-Tree
If T is a B-tree of order m and height h, and where d = ém/2ù  and let n be the number of elements in T.
2dh-1  - 1 < n < mh –1
logm(n+1) < h < logd((n+1)/2) + 1
So if the B-tree is of order 200 and height 3 and has at least 19,999 elements, and one of the same order and height 5 has at least 2*108 –1 elements.
Order is determined in practice from the block size.
There is no advantage to using a node size smaller than the disk block size.
In practice (Operating Systems) m is typically 256-1024
All our examples will have small m. (3 or 7)

Merge Discussion
The worst case for a deletion from a B-tree of height h is when the merge takes place all the way up the tree and we get an element from a nearest sibling.
The worst case disk access count is 3h
h reads to find the leaf
h-1 reads to get nearest siblings at levels (2-h)
h-2 writes of merged nodes
3 writes for the modified root, and 2 level 2 nodes
B+ Trees
The previous discussion mentioned that B-Trees are universally used to implement large-scale disk-based file systems.
We lied. What is implemented is a variant of the B-tree called B+ tree.
When greater efficiency is required, a slightly more complicated variant known as the B* tree is used.
The most significant difference between a B+ and B trees is that B+ trees store records only at the leaf nodes. The internal nodes only store key values to help guide search.
The big advantage of B+ trees is that the keys are small enough so that the top levels of the tree can remain in main memory and do not require any disk accesses.
B+ Trees Searching/ Insertion
Searching in a B+ tree is nearly identical to searching a B tree except that every search must go to the leaf.  The internal nodes are just keys.
Inserting a record into a B+ tree is similar to B-tree insertion.
First the appropriate leaf is found.
If it is not full the record is added
If full it is split and the copy of the least valued key in the right node is promoted.
Note the promotion may cause the parent to be split
